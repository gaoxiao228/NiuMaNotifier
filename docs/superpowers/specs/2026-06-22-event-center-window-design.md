# 事件中心独立窗口设计

## 背景

当前事件中心作为设置页中的一个面板存在，只在用户打开“设置 > 事件中心”时连接 `/api/v1/events/stream`。这个位置更像调试工具，不适合作为状态栏常驻应用中的独立观察入口。

本次设计将事件中心从设置页移出，改为由状态栏菜单打开的独立窗口。关闭事件中心窗口后停止监听，下次打开重新从空列表开始接收实时事件。

## 目标

- 从设置页移除事件中心入口和内嵌面板。
- 不在主状态页增加事件中心入口。
- 在状态栏菜单中增加“显示事件中心”选项。
- 使用菜单勾选状态标记事件中心窗口当前是否显示。
- 事件中心作为独立窗口展示实时 `NiumaEvent`。
- 事件中心窗口关闭后停止 SSE 监听并清空本次窗口内事件。
- 下次打开事件中心时重新连接 `/api/v1/events/stream`，只展示打开后收到的新事件。
- 新增界面和状态栏文案必须覆盖简体中文、繁体中文、英语、日语、韩语、德语。

## 非目标

- 不新增历史事件查询。
- 不新增搜索、过滤、导出、复制等高级能力。
- 不改变 `/api/v1/events/stream` 的实验性语义。
- 不改变主状态聚合逻辑。
- 不改变通知历史、插件管理和通知设置页面。
- 不引入新的 workspace crate。

## 推荐方案

采用独立 Tauri 窗口和独立前端入口。

新增一个窗口 label，例如 `event-center`。该窗口只渲染事件中心，不复用主窗口中的设置页、插件管理和主状态逻辑。前端新增入口文件，例如 `src/eventCenterWindow.ts`，复用现有事件中心模块：

- `createEventCenterRuntime`
- `renderEventCenter`
- `setEventCenterItemExpanded`
- `getLocalApiUrl`
- `i18n`

这种方案边界清晰，避免继续扩大 `src/main.ts` 的职责，也能让事件中心窗口的打开、关闭、监听生命周期保持独立。

## 备选方案

### 方案 A：独立窗口 + 独立前端入口

这是推荐方案。

优点：

- 主窗口、设置页、事件中心职责清楚。
- 关闭窗口即可自然释放 SSE、DOM 状态和事件缓存。
- 后续维护成本低。

缺点：

- 需要新增 Tauri 窗口管理逻辑。
- 需要调整前端构建入口。

### 方案 B：独立窗口 + 复用主前端入口

事件中心窗口加载同一个前端包，通过 query 或 hash 判断渲染模式。

优点：

- 构建配置改动较少。

缺点：

- `src/main.ts` 已经承担主状态、设置、插件、通知、语言同步等逻辑，继续加入窗口模式会增加耦合。
- 事件中心生命周期容易和主窗口状态相互影响。

### 方案 C：仍使用主窗口内部视图

状态栏点击后打开主窗口并切换到事件中心视图。

优点：

- 改动最小。

缺点：

- 不符合独立窗口要求。
- 主状态页不应增加入口，内部视图会让产品边界变模糊。

## 窗口生命周期

状态栏“显示事件中心”菜单项负责打开和关闭事件中心窗口。

规则：

- 窗口不存在时，点击菜单项创建 `event-center` 窗口并显示。
- 窗口存在但不可见时，点击菜单项显示并聚焦窗口。
- 窗口可见时，点击菜单项关闭或销毁窗口。
- 用户点击窗口关闭按钮时，允许事件中心窗口关闭或销毁。
- 事件中心窗口关闭时，前端 `beforeunload` 调用 `eventCenterRuntime.stop()`。
- 事件中心窗口关闭后清空本次事件缓存。
- 再次打开窗口时重新创建前端运行态，重新连接 SSE。
- 主窗口关闭仍保持当前隐藏到状态栏的行为。
- 应用真正退出时，主窗口和事件中心窗口都退出。

建议关闭事件中心窗口时销毁窗口，而不是长期隐藏。这样最符合“关闭后停止监听”的需求，也避免保留旧事件列表造成误解。

## 状态栏菜单

状态栏菜单建议顺序：

```text
显示 NiumaNotifier
显示事件中心
当前状态：运行中
语言
退出 NiumaNotifier
```

“显示事件中心”使用可勾选菜单项：

- 事件中心窗口可见时勾选。
- 事件中心窗口关闭后取消勾选。
- 通过状态栏打开或关闭时同步勾选状态。
- 通过窗口关闭按钮关闭时同步勾选状态。
- 语言切换时刷新菜单文案，但保持勾选状态。

托盘状态刷新线程仍只负责状态相关文案刷新和语言文案刷新，不应主动改变事件中心窗口可见状态。

## 前端行为

事件中心窗口前端只负责事件中心视图：

- 初始化当前语言。
- 渲染窗口外壳和事件中心列表。
- 打开窗口后启动 `createEventCenterRuntime.start()`。
- 关闭窗口或页面卸载时调用 `stop()`。
- 点击事件行时展开或收起 JSON 详情。
- 监听 `niuma-language-changed`，语言变化后重新渲染窗口文本。

窗口 UI 保持工具窗口风格：

- 顶部展示标题和连接状态。
- 主体展示事件列表。
- 每条事件展示事件类型、工具、项目、摘要和时间。
- 展开后展示完整 JSON。

不新增历史、搜索、过滤、导出等功能，避免过度设计。

## 国际化

前端继续使用 `src/i18n.ts` 中的事件中心文案。设置页移除事件中心后，不再使用设置页中的事件中心描述文案。

Rust 状态栏需要新增事件中心菜单文案：

| 语言 | 文案 |
| --- | --- |
| 简体中文 | 显示事件中心 |
| 繁体中文 | 顯示事件中心 |
| 英语 | Show Event Center |
| 日语 | イベントセンターを表示 |
| 韩语 | 이벤트 센터 표시 |
| 德语 | Ereigniszentrum anzeigen |

## 错误处理

- 获取 Local API URL 失败时，事件中心窗口展示错误文本，不退出窗口。
- SSE 断开时展示断开状态和错误文本。
- 收到非法事件 payload 时展示错误文本，并继续保持窗口可用。
- 重复事件 ID 不重复追加。

这些行为延续现有 `eventCenterRuntime` 的处理方式。

## 测试范围

Rust 单元测试：

- 状态栏 labels 包含事件中心文案。
- 不同语言下事件中心状态栏文案正确。
- 主窗口关闭策略不影响事件中心窗口关闭策略。
- 状态栏菜单刷新时保留事件中心勾选状态。

前端测试：

- 设置页不再渲染事件中心导航项。
- 事件中心窗口空列表、连接中、已连接、断开状态可正确渲染。
- 事件列表按收到顺序追加。
- 点击事件项只展开或收起对应 JSON 详情。
- 语言变化后事件中心窗口重新渲染文案。

构建和验证：

- 运行现有前端测试。
- 运行 `cargo test`。
- 运行 `npm run build`。
- 如新增 Tauri 窗口配置，至少执行一次桌面端构建或等价检查，确认窗口入口被正确打包。
